## P0

### P0-1 · `/business/content` crash systématique après chargement ✅vérifié
- `components/business/ContentPlanner.tsx:810` : `useState('')` placé APRÈS le `if (loading) return` de la ligne 427 (loading init `true`). 3ᵉ occurrence de la classe « hook après early-return » (bloodtest 2026-05-02, dashboard 2026-05-10).
- **Symptôme** : skeleton → fin du chargement → « Rendered more hooks » → page blanche, 100 % des visites.
- **Fix** : remonter le `useState` avec les autres (L83-114). Effort **S**.

### P0-2 · Roadmap crash à la création de la 1ère phase (ou suppression de la dernière) ✅vérifié
- `components/roadmap/RoadmapCalendar.tsx:244` : `useMemo` APRÈS `if (!phases.length) return null` (L158). Composant rendu inconditionnellement (`roadmap/page.tsx:420`).
- **Symptôme** : `phases` passe de `[]` à `[...]` (ou l'inverse) → crash « This page couldn't load ».
- **Fix** : déplacer le `useMemo` au-dessus de L158 (il gère déjà le cas vide). Effort **S**.

### P0-3 · AthleteContext : aucun refetch au réveil → skeleton infini en cascade ✅vérifié
- `contexts/AthleteContext.tsx:134-145` : SWR `revalidateOnFocus:false` (correct) mais **rien** n'écoute `coach:wake` dans le provider (grep confirmé : seules les pages consomment `useRefetchOnResume`).
- **Symptôme** : fetch initial gelé → `loading` reste `true` → `/athletes` (`AthletesList.tsx:179`), `/dashboard` (`DashboardPage.tsx:310`, dont le `swrKey` reste `null`) et `athletes/[id]/layout.tsx:66-75` en skeleton permanent. C'est le signalement exact du coach.
- **Fix** : `useRefetchOnResume(() => { mutate() }, isLoading)` dans `AthleteProvider`. Effort **S**.
