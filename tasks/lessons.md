# Lessons Learned

[2026-05-10] | Rules of Hooks RÉPÉTÉ — useMemo dashboard mrr/activePayingAthletes placé après `if (loading) return` → "This page couldn't load" sur preview. C'EST LA MÊME ERREUR QUE [2026-05-02] sur bloodtest. RÈGLE RENFORCÉE : avant chaque commit qui ajoute un hook (useMemo/useCallback/useEffect/useState/useSWR/useRouter), CHERCHER `if (` ET `return` dans le fichier au-dessus du hook, et confirmer que TOUS les early-returns sont APRÈS le hook. Pas d'exception. Si on doit conditionner, le faire DANS le useMemo (return null/empty), pas autour.

[2026-05-02] | OpenFoodFacts CGI search (`/cgi/search.pl`) déprécié, retourne page HTML "temporarily unavailable" → JSON.parse fail silencieux → 0 résultat | La recherche d'aliments par nom (NutritionScreen FoodSearchModal, /aliments page, components/nutrition/FoodSearch) tombait à 0. RÈGLE : pour les APIs publiques tierces, surveiller les changements d'endpoint (les CHANGELOG des projets, les status pages). Nouvel endpoint OFF : `https://search.openfoodfacts.org/search` (ElasticSearch, retourne `hits` au lieu de `products`, `brands` array au lieu de string CSV). Le barcode product API `https://world.openfoodfacts.org/api/v2/product/{barcode}.json` lui n'a pas changé. Toujours ajouter un User-Agent header sur les fetch externes (OFF le requiert maintenant).

[2026-05-02] | useMemo placé après early-return = page crash silencieux | Sur bloodtest/page.tsx, j'ai ajouté `const analysisEtaMs = useMemo(...)` APRÈS les `if (loading) return` et `if (!enabled) return`. React appelle un nombre différent de hooks selon le state → crash "This page couldn't load". RÈGLE : TOUS les hooks (useMemo/useCallback/useEffect/useState) doivent être appelés AVANT tout early-return d'un composant. Pas d'exception. Si tu ajoutes un hook dans un composant existant, vérifie qu'il est placé avant les `if (...) return`.

[2026-04-29] | `git checkout` ne persiste pas entre les Bash tool calls | Le cwd reset entre commandes mais aussi le HEAD si on ne lock pas la branche. Edits faites sur `main` au lieu de `feature/meal-variants` à cause d'un checkout dans un Bash isolé. RÈGLE : pour un travail multi-commits sur une branche feature, vérifier `git branch --show-current` AU DÉBUT de chaque session de Bash, et faire le checkout dans la même Bash que les premiers edits si possible. Sinon, accepter de stash/pop entre branches.

[2026-04-29] | Parsers meals_data legacy strippent les variants silencieusement | Tous les helpers `parseMealsArray`/`parseMealsData`/`toMeal` dans templates et nutrition pages ne gardaient que `{ foods }` et `pre_workout`/`time`. Quand un repas a `{ variants: [...] }` au lieu de `foods`, ces parsers le convertissaient en `{ foods: [] }` → variantes perdues au prochain save. RÈGLE : tout parseur de meal qui round-trip vers le DB doit préserver `variants` quand présent (test : `Array.isArray(m.variants) && m.variants.length > 0` AVANT le check `m.foods`).

[2026-03-31] | Supabase v2 type incompatibilities with `.catch()` on query builders | The newer `@supabase/supabase-js` returns `PostgrestFilterBuilder` (not a Promise) from `.insert()` etc., so `.catch(() => {})` fails at type-check. Remove the `.catch()` or use try/catch instead.

[2026-03-31] | Pre-existing type errors block build | Always run `npx tsc --noEmit` to verify new code doesn't introduce errors, but be aware pre-existing errors in other files may also surface during `npm run build`.

[2026-03-31] | Data leak — unfiltered queries | Any query on shared tables (daily_reports, notifications, etc.) MUST filter by coach's athlete user_ids. Never query without a coach/user scope filter.

[2026-03-31] | Object references in useCallback/useEffect deps cause re-render loops | Never use context-derived objects (selectedAthlete, user) as dependency array values. Use primitive fields instead (selectedAthlete?.id, user?.id). Also memoize context provider values with useMemo to prevent unnecessary consumer re-renders.

[2026-03-31] | setLoading(true) without try/finally leaves UI stuck | Always wrap the body of async functions that call setLoading(true) in a try/finally block with setLoading(false) in finally.

[2026-03-31] | select('*') wastes bandwidth and slows queries | Always use explicit column lists in Supabase .select() calls. Add .limit() to any query that could return many rows.

[2026-04-03] | Heavy JSON columns (meals_data) kill list view performance | Don't fetch large JSON columns in list queries. Only fetch them on demand.

[2026-04-03] | Sequential queries avoidable via context data | Use AthleteContext data (already loaded) instead of re-querying the DB for athlete info.

[2026-04-13] | NEVER wrapper le fetch global de Supabase | Le wrapper AbortController + fake 408 Response a cassé le refresh de token auth. Toutes les requêtes retournaient vide. Le client Supabase doit rester VANILLA. Pour les timeouts Safari, utiliser useRefetchOnResume (hook composant).

[2026-04-13] | revalidateOnFocus:true sur SWR = cascade de re-fetch | Chaque switch d'onglet/app relançait 3 requêtes AthleteContext. Toujours mettre revalidateOnFocus:false.

[2026-04-13] | setLoading(true) inconditionnel = skeleton flash sur refetch | Pattern correct : `if (!data.length) setLoading(true)` — skeleton uniquement au premier chargement, pas sur les refetch.

[2026-04-13] | useCallback deps sur user (objet) au lieu de user?.id (string) | L'objet user change de référence à chaque auth event, relançant tous les useCallback/useEffect qui en dépendent.

[2026-04-13] | Toujours npm run build avant push | Plusieurs commits pushés sans vérification ont cassé le site en prod.

[2026-04-20] | Ouvrir la console AVANT de patcher un bug | J'ai passé 2h à patcher des symptômes (wake handler, timeouts, watchdog) pour le bug tab-switch avant que le user ouvre enfin la console qui montrait directement `React error #418` puis `@supabase/gotrue-js: Lock was not released within 5000ms`. Règle : premier réflexe sur un bug client = demander console + Network, pas deviner.

[2026-04-20] | Hydration mismatch (React #418) = UI gelée en skeleton | Lire localStorage/window/typeof document pendant le render initial fait diverger SSR et client. React bail, UI reste dans l'état SSR = skeleton infini. Pattern correct : initial state identique SSR+client (null, false, defaults), lire les APIs browser dans useEffect post-hydration. Fichiers typiques : Context providers, layouts, useState lazy initializers.

[2026-04-20] | Safari tab background + Supabase auto-refresh = orphan lock | @supabase/auth-js wrap chaque opération auth dans un mutex navigator.locks. Safari gèle les fetch dans les tabs en arrière-plan. Si auto-refresh fire en background → fetch gelé EN tenant le lock → orphan. Fix officiel : `supabase.auth.stopAutoRefresh()` sur hidden, `startAutoRefresh()` sur visible. PAS de await, PAS de getSession/getUser dans le handler. Doc : https://supabase.com/docs/reference/javascript/auth-startautorefresh

[2026-04-20] | getSession() ne fait AUCUN call réseau | Contrairement à ce qu'on peut croire, `supabase.auth.getSession()` lit juste le localStorage. Pour vraiment tester la connexion Supabase, utiliser `getUser()` (hit /auth/v1/user) ou un plain fetch REST. MAIS : getUser() acquiert aussi le lock → risque d'orphan. Pour un vrai ping sans lock, plain `fetch(supabaseUrl + '/rest/v1/')`.

[2026-04-20] | Chercher les issues GitHub de la lib AVANT de patcher un bug profond | Le bug Safari + Supabase était déjà documenté dans supabase-js#2111, #1594, #2013 avec le fix officiel. J'ai empilé 4 patches symptomatiques (no-op lock, wake ping, watchdog reload, reload on timeout) avant de faire la recherche. Règle : dès qu'un bug touche à une lib tierce (auth, fetch, lock), grep les issues GitHub de la lib en priorité absolue.

[2026-04-20] | Watchdog reload-on-stuck MASQUE les bugs et peut les aggraver | J'avais ajouté un watchdog qui reload la page si loading stuck >10s. Ça donne l'illusion que ça marche mais : (1) le user voit un reload inexpliqué, (2) le nouveau client peut re-deadlock instantanément, (3) ça masque la vraie cause racine. Règle : fix la cause, jamais un watchdog reload en prod.

[2026-04-20] | Preview Vercel ≠ prod pour tester RSC | Les preview URLs Vercel ont Deployment Protection (SSO) activé → les prefetches RSC échouent en 401 → Next.js fallback en full page reload = nav lente. Sur prod avec custom domain, pas de SSO, RSC marche. Pour tester vraiment la nav App Router, utiliser la prod (ou désactiver SSO sur preview).

[2026-04-20] | Colonnes DB : vérifier les SELECT contre le schéma réel | Une query `select('id, athlete_id, coach_id, reason, status, ...')` sur `cancellation_requests` retournait 400 silencieusement pendant des semaines parce que la colonne est `coach_note`, pas `reason`. Les 400 Supabase ne throw pas — ils retournent `{ data: null, error }` et le code continue avec data vide. Toujours grep les SQL migrations pour vérifier les colonnes existantes.

[2026-04-20] | Empty string '' vers colonne DATE Postgres = 400 | `date_naissance: ""` dans un UPDATE fail avec `22007 — invalid input syntax for type date`. Toujours normaliser les champs optionnels en `null` avant un update Supabase : `for (const k of Object.keys(data)) if (data[k] === '') data[k] = null`.

[2026-04-20] | Supabase client MUST be module-scope singleton | Si createClient() est appelé dans le body d'un component et retourne un nouveau client à chaque fois, chaque render crée un holder du lock auth sans libérer le précédent = orphan garanti. Pattern correct : `let client = null; export function createClient() { if (!client) client = createBrowserClient(...); return client }`. Notre code l'a déjà mais c'est fragile, à garder en tête.

[2026-04-20] | Les erreurs "Fetch API cannot load ... due to access control checks" sur preview Vercel = SSO, pas CORS | Pattern spécifique aux Vercel preview URLs avec Deployment Protection. Pas réparable côté code sauf à désactiver le SSO. Ne pas chercher des heures côté CORS ou headers — c'est juste Vercel.

[2026-04-21] | Une branche feature non mergée = fix perdu | Le fix "empty string → null" avait été fait sur `feature/debug-bilan-save-error` qui n'a jamais été mergée vers develop. 6h plus tard, user redécouvre le même bug. Règle : vérifier l'état des branches ouvertes à la fin d'une session (`gh pr list --author @me`) et soit merger, soit fermer avec raison. Sinon les fixes meurent.

[2026-04-21] | Toast d'erreur générique masque la cause | `toast('Erreur lors de la sauvegarde', 'error')` empêche le user de voir le vrai message Postgres. Toujours : `toast(\`Erreur: ${error.message}\`, 'error')` + `console.error('[ctx]', error, 'payload:', data)`. Le user peut copier/coller la vraie erreur au lieu de deviner.

[2026-04-24] | Convention nom/name mélangée dans la DB = bugs silencieux | Certaines tables utilisent `nom` (training_templates, nutrition_templates) et d'autres `name` (onboarding_workflows). Un SELECT avec le mauvais column name retourne 400 silencieusement, l'UI affiche une liste vide. Toujours vérifier le schéma (grep les autres queries sur la même table) avant d'écrire un SELECT. Ajouter systématiquement `if (error) console.error('[ctx]', error)` sur toutes les queries Supabase.

[2026-04-24] | Insert Supabase client-side pour les données d'un AUTRE user = RLS silent fail | AddAthleteForm insérait `athlete_onboarding` avec `athlete_id = new_user.auth_uid`. RLS bloquait car coach's `auth.uid()` ≠ `athlete_id` de la row. Sans `if (error)`, l'await résout OK, personne ne sait. Résultat : depuis des mois, zéro row créée. Règle : tout insert/update Supabase qui touche des données d'un AUTRE user DOIT passer par une API route serveur avec service role. Et : jamais de `await supabase.from(...).insert(...)` sans lire `error`.

[2026-04-24] | "This page couldn't load" Safari = crash WebContent, pas erreur réseau | Ce message apparaît quand le process de rendu JS crash (infinite loop, memory, Rules of Hooks violation). PAS une erreur réseau. Ouvrir la console AVANT de cliquer sur l'action qui déclenche le crash pour voir l'erreur JS réelle (souvent un React error code).

[2026-04-24] | React error #300 = Rules of Hooks violation ou setState en render | "Too many re-renders". Cause classique : `useMemo/useCallback/useEffect` placé APRÈS un `return` conditionnel — le nombre de hooks change entre renders. RÈGLE : **tous les hooks AVANT tout return**, même les returns précoces. Le linter react-hooks catche normalement mais peut être désactivé par eslint-disable.

[2026-04-28] | AppState 'active' sync skip par cooldown = bug "force-quit pour rafraîchir" | useStepsSync avait `if (now - lastSync < 30min) return` qui s'appliquait AUSSI à l'AppState 'active' handler. Conséquence : passer l'app en background 5min puis revenir ne déclenchait aucune resync. Force-quit l'app re-init lastSync à 0 → seul moyen de rafraîchir. RÈGLE : différencier "interval timer" (respecte cooldown) et "event trigger" (force=true). Sur foreground, toujours bypasser le cooldown (avec debounce court 2s pour éviter les multiples 'active' iOS).

[2026-04-28] | HealthKit/Health Connect peut retourner 0 != "0 pas réels" | `result.sumQuantity` peut être null si HealthKit n'a pas encore les données (autorisation partielle, device pas encore syncé). Le code faisait `quantity ?? 0` puis le check `if (steps == null) return` ne filtrait pas ce 0 → écrasait la vraie valeur en BD. RÈGLE : pour toutes les sources externes (HealthKit, Health Connect, APIs tierces), distinguer "pas de données" (null) de "valeur 0 réelle". Ne jamais default à 0 dans le code de read si l'absence de donnée est sémantiquement différente.

[2026-04-28] | Sauvegarder une valeur < l'existante = perte silencieuse | `upsertDailyTracking(athleteId, date, { steps: 0 })` quand la BD a déjà steps=5000 → écrase à 0. Ajouter une garde côté client (`if (steps > 0 || existing === 0)`) avant le write. Idéalement aussi côté DB (trigger ou RLS) pour les valeurs cumulatives qui ne devraient jamais décroître au cours d'une journée.

[2026-04-28] | catch silencieux qui reset l'UI à 0 = bug invisible | `} catch { setStepsInput('0') }` sur une erreur réseau force l'UI à 0 sans retry, sans log, et écrase potentiellement une valeur déjà mise par un autre hook. RÈGLE : tout catch doit logger l'erreur (`console.error('[ctx]', e)`), ne PAS reset une valeur existante par défaut, et idéalement laisser l'UI dans son état précédent (functional updater `setX(prev => prev)` ou skip le set).

[2026-04-29] | Permissions-Policy `camera=()` bloque getUserMedia partout | Le header `Permissions-Policy: camera=(), microphone=(), geolocation=()` dans next.config.ts désactive ces APIs pour TOUTE l'origine, y compris `self`. Une feature qui utilise getUserMedia/getDisplayMedia plante en prod alors qu'elle marche en dev (où le header n'est pas appliqué côté next dev). RÈGLE : toujours auditer les headers de sécurité quand on ajoute une feature qui touche aux media APIs. Format correct : `camera=(self), microphone=(self), display-capture=(self)`.

[2026-04-29] | MediaRecorder.onstop race quand stop déclenché par browser-end ou hard-cap | Pattern naïf : `stopRecording = () => { recorder.addEventListener('stop', finalize); recorder.stop() }`. Si `recorder.stop()` a déjà été appelé ailleurs (track 'ended' du screen share, hard-cap timer), au moment où l'utilisateur clique Stop dans l'UI, l'event a déjà fired → listener jamais attaché à temps → UI bloquée à "Traitement…" et vidéo perdue. FIX : un seul `recorder.onstop` configuré à la construction du recorder, qui cache le résultat dans un ref. `stopRecording()` returns le résultat caché si présent, sinon installe un waiter que `onstop` resolve. RÈGLE : pour les MediaRecorder, toujours configurer onstop ONCE à la construction et coordonner via refs/promises, jamais ajouter le listener au moment du stop.

[2026-04-29] | requestAnimationFrame loop : capturer l'ID externe au draw | `let id = 0; const draw = () => { ...; id = requestAnimationFrame(draw) }; id = requestAnimationFrame(draw); return id` — le caller capture l'ID INITIAL, pas les IDs suivants générés par chaque tick. cancelAnimationFrame(idInitial) ne fait rien après la 2e frame. FIX : utiliser un ref/objet partagé `const idRef = { current: 0 }` mis à jour par le draw lui-même, et exposer une fonction `stop()` au lieu de l'ID brut.

[2026-04-29] | Storage upload + DB metadata = vérifier que le fichier existe | Risque de path-phishing : un coach authentifié peut envoyer une video_path arbitraire à l'API et créer un retour qui pointe vers le fichier d'un autre coach. FIX : valider côté serveur (1) le path commence par `${user.id}/`, (2) l'objet existe vraiment dans le bucket via `storage.list(folder, { search: id })`. Sans ça, RLS ne suffit pas car l'API est appelée avec service role.

[2026-04-29] | DB-first puis storage cleanup pour les deletes | Pattern naïf : delete les fichiers Storage puis le row DB. Si le DB delete échoue (RLS, transient), les fichiers sont gone mais le row pointe vers du vide → player 404 permanent. RÈGLE : DELETE le row DB d'abord, PUIS best-effort cleanup Storage (catch + warn, pas fail).

[2026-06-15] | Ne JAMAIS écrire `coach_id: coach.id` dans un INSERT/UPDATE — `coach.id` = `coach_profiles.id` (row du profil), pas `auth.users.id` | Quasi toutes les tables avec un `coach_id` (RLS = `coach_id = auth.uid()`) attendent l'auth user id. `coach.id` est un UUID différent (row id de la table `coach_profiles`). Utiliser `coach.id` à la place provoque un **insert silencieusement bloqué par RLS** — pas d'erreur SQL claire, juste un retour vide (effet visible : "Nouveau X" ne fait rien). Vu sur QuestionnaireTemplatesList le 2026-06-15. RÈGLE : pour toute écriture coach-owned, importer `useAuth` et utiliser `user.id`. `coach` ne sert que pour afficher le profil (nom, plan, stripe state). À ajouter à la code review systématique : `grep -n "coach\.id" components/` ne doit retourner aucun INSERT/UPDATE/payload.

[2026-06-14] | Incident Cloudflare/Supabase → transactions PostgREST orphelines bloquent tout le pool ensuite | Quand Supabase a un pic d'erreurs 522/504 (Cloudflare timeout) ou un blip Postgres, les clients (apps mobiles, web) qui étaient en plein milieu d'une requête se déconnectent côté TCP. Côté serveur, PostgREST a déjà ouvert un `BEGIN` dans une connection ; le client coupe avant d'envoyer la query ou le `COMMIT` ; Postgres garde la session en `idle in transaction` indéfiniment. Au bout de quelques minutes, le pool (max_connections=60 sur le tier actuel) sature → TOUTES les nouvelles requêtes timeout, y compris le SQL Editor du dashboard et le MCP. SYMPTÔME app mobile : "charge dans le vide", auth refresh stuck, gap de 14s+ avant les premiers fetch. DIAGNOSTIC : `SELECT pid, state, now()-state_change AS idle_for, application_name FROM pg_stat_activity WHERE state = 'idle in transaction' AND application_name = 'postgrest'`. FIX URGENCE : `SELECT pg_terminate_backend(pid) FROM pg_stat_activity WHERE state = 'idle in transaction' AND application_name = 'postgrest' AND now() - state_change > interval '30 seconds'`. FIX PERMANENT À FAIRE : configurer `idle_in_transaction_session_timeout` à 60s côté Postgres (Supabase Dashboard → Database → Postgres Config), pour que ces orphelines soient killées automatiquement et plus jamais re-saturer le pool après un incident upstream.

[2026-07-18] | Audit complet : la cause n°1 des "bugs de chargement" côté coach n'était PAS le serveur (logs Vercel quasi propres) mais 2 mécanismes CLIENT invisibles dans les logs | (1) `AthleteContext` avait `revalidateOnFocus:false` (correct) mais AUCUN `useRefetchOnResume` dans le provider : un fetch initial gelé (Safari onglet en arrière-plan, blip Supabase) laissait `/athletes`, `/dashboard` et le header athlète en skeleton PERMANENT. Le pattern wake-refetch avait été appliqué à 23 pages mais jamais au provider central qui alimente les 3 écrans principaux. RÈGLE : tout provider SWR qui alimente un écran gaté par `if (loading)` DOIT avoir `useRefetchOnResume(() => mutate(), isLoading)`, pas seulement les pages. (2) 3 crashs Rules of Hooks non attrapés (ContentPlanner, RoadmapCalendar, WeightChart) = page blanche sans retry, ABSENTE des logs serveur car crash client. RÈGLE : toujours avoir un `app/(app)/error.tsx` — un crash client sans error boundary est indistinguable d'un "chargement infini" pour le user ET pour le debug.

[2026-07-18] | Objet `user` re-créé à chaque TOKEN_REFRESHED (~1x/h) → refetch storm sur ~20 pages avec deps [user] | `onAuthStateChange` faisait `setUser({id, email})` = nouvelle référence même si l'id est inchangé. Toutes les pages avec `}, [user])` refetchaient d'un coup chaque heure, et pendant un blip Supabase ces requêtes s'empilaient et aggravaient la saturation du pool. FIX : `setUserStable` qui préserve la référence si id+email inchangés (`setUser(prev => prev?.id===u.id && prev?.email===u.email ? prev : u)`). RÈGLE déjà connue (2026-04-13 "deps user objet") mais la source amont (setUser dans le context) n'avait jamais été corrigée — corriger à la SOURCE vaut mieux que patcher 20 deps arrays.

[2026-07-18] | Routes API service-role : verifyAuth prouve QUI appelle, pas SUR QUOI il agit | Plusieurs routes (stripe cancel, instagram thread/send, connect-complete) authentifiaient le caller mais n'vérifiaient pas que la ressource ciblée (subscriptionId, conversation_id, coachId) lui appartenait → IDOR en service role. Webhook Meta n'avait AUCUNE vérification de signature X-Hub-Signature-256 → injection de faux DMs. RÈGLE : sur toute route service-role, après verifyAuth, TOUJOURS re-vérifier l'ownership de la ressource (`SELECT ... WHERE id=? AND owner=auth.uid()`) avant d'agir. Sur tout webhook tiers, vérifier la signature HMAC avant de traiter le body. Tokens tiers (IG/FB) : `lib/api/crypto.ts` existait mais n'était JAMAIS appelé — un module de sécurité mort est pire qu'absent (fausse assurance).
