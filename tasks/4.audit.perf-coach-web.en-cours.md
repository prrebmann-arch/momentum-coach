# 🔴 CRITIQUES — à fixer en priorité

## C1. Stripe Webhook → risque de DOUBLE CHARGE

**`app/api/stripe/webhook/route.ts`** : pas d'idempotence. Le check `stripe_audit_log` log mais ne **previent pas** la ré-exécution business logic. Si Stripe retry un event (ex: timeout réseau), l'athlète est facturé deux fois.

**Fix** : check + insert dans `stripe_processed_events` (UNIQUE event id) **avant** le handler.

**Impact** : critique sécurité financière, pas perf.

## C2. FontAwesome 190 KB pour 13 icônes

**`app/layout.tsx:7-10`** importe 4 CSS bundles complets (`fontawesome.min.css`, `solid.min.css`, `regular.min.css`, `brands.min.css`). 258 usages `fa-solid`, 1 seul `fa-brands` (youtube), 1 seul `far` (regular cercle).

```
75 KB (fontawesome.min) + 16 KB (brands) + 0.6 KB (regular) + 0.6 KB (solid) = ~92 KB CSS
+ 4 woff2 fichiers polices = ~100 KB additionnels
```

**Fix** :
- Court terme : retirer `regular.min.css` et `brands.min.css` (1 seul usage → remplacer le YouTube par SVG inline ou switch à `fa-solid fa-youtube`)
- Long terme : migrer à `lucide-react` (tree-shakable, ~3 KB pour 13 icônes)

**Impact** : -180 KB sur le 1er load = **réduction visible de TTFB sur 4G**

## C3. `RecorderContext` (415 lignes) chargé sur TOUTES les routes

**`app/(app)/layout.tsx:9-12`** : `RecordingPill`, `LiveCamBubble`, `RetourFinalizeModal` + `RecorderProvider` montés sur dashboard, athlètes, aliments, exercices, etc. — alors qu'ils ne servent QUE quand le coach enregistre un retour vidéo.

**Fix** : `next/dynamic({ ssr: false, loading: () => null })` sur les 3 composants UI. Le `RecorderProvider` peut rester (il est léger sans son hook actif), mais les composants visuels qui pèsent le JS bundle sont à différer.

**Impact** : -50-80 KB sur 1er load + parsing plus rapide

## C4. Supplements page : N+1 + heavy JSON

**`app/(app)/athletes/[id]/supplements/page.tsx`** :
- ligne 337-342 : fetch `nutrition_plans` avec **`meals_data` JSONB** (10-50 KB par plan) **juste pour compter les meals**
- ligne 627-668 : pour chaque assignment, `getAssignmentHistory()` loop sur 14 jours en filtrant les logs en JS — avec 20 assignments = 280 `array.filter()` calls
- ligne 672-694 : compliance grid recalculé à chaque tab change, pas de `useMemo`

**Fix** : 
- Remplacer `meals_data` par un COUNT serveur ou stocker `meals_count` dénormalisé
- Pré-indexer les logs par `assignment_id` dans un Map au mount
- `useMemo` sur compliance grid avec `[tab, logs]`

**Impact** : -2 s sur load supplements + tab switch fluide

## C5. Stripe `/api/stripe/cron` — timeout 120s

**`app/api/stripe/cron/route.ts:85-158`** : pour chaque coach, séquentiel `invoiceItems.create` (×2) + `invoice.create` + `finalize` + `pay`. Avec 50+ coachs = ~100s minimum. Vercel Functions timeout = 60s par défaut, 300s max.

**Fix** : 
- `Promise.all` sur les batchs de 5-10 coachs en parallèle
- Configurer `export const maxDuration = 300` dans le route file
- Si toujours trop long : split en 2 crons (préparation + exécution)

**Impact** : évite les facturations skippées en prod

## C6. Indexes DB manquants sur RLS hot paths

Existant : `sql/perf_indexes.sql` (27 indexes). Manquants identifiés :

```sql
-- Hot paths RLS-via-subquery
CREATE INDEX IF NOT EXISTS idx_daily_tracking_user_id ON daily_tracking(user_id);
CREATE INDEX IF NOT EXISTS idx_menstrual_logs_athlete_id ON menstrual_logs(athlete_id);
CREATE INDEX IF NOT EXISTS idx_questionnaire_responses_athlete_id ON questionnaire_responses(athlete_id);
CREATE INDEX IF NOT EXISTS idx_exercise_settings_user_id ON exercise_settings(user_id);
CREATE INDEX IF NOT EXISTS idx_bilan_retours_athlete_id ON bilan_retours(athlete_id);

-- Composites pour ORDER BY après filter
CREATE INDEX IF NOT EXISTS idx_workout_logs_coach_created ON workout_logs(coach_id, created_at DESC);
CREATE INDEX IF NOT EXISTS idx_payment_history_coach_created ON payment_history(coach_id, created_at DESC);

-- Partial pour notifications non-lues
CREATE INDEX IF NOT EXISTS idx_notifications_unread ON notifications(user_id, created_at DESC) WHERE read = false;
```

**Impact** : sur 10K+ rows, transition seq-scan → index scan = **10-100× plus rapide** sur les queries dashboard / athlète / notifs.

⚠️ **Vérifier en SQL Editor** que ces indexes n'existent pas déjà avant de les créer (voir `\d+ <table>`).

# 🟠 IMPACT FORT, EFFORT MODÉRÉ

## H1. `getPageCache` / `setPageCache` no-op encore appelés

Toujours dans `nutrition`, `training`, `apercu`, `bilans`. Lessons.md flag déjà ça mais c'est jamais nettoyé. Risque : `setLoading(true)` skippé si cache initial → pas de feedback UI.

**Fix** : remplacer par `useState(() => null)` + SWR.

## H2. `BusinessDashboard` 7 queries en mount + waterfall

`components/business/BusinessDashboard.tsx:343` : Promise.all OK pour 6 queries, mais `weekly_objectives` (ligne 372) est **après** dans un useEffect séparé → waterfall de 1-2s.

**Fix** : intégrer `weekly_objectives` dans le Promise.all initial.

## H3. `InstagramAnalytics` sync N+50 sequential

`components/business/InstagramAnalytics.tsx:247-308` : pour 50 reels, 50 calls Graph API en série → **30-60s** d'attente. Plus le `JSON.parse` non-mémoïsé dans render loop ligne 258-265.

**Fix** : 
- `Promise.all` avec batching (10 par batch pour respecter rate limit Graph)
- Cache server-side dans `ig_reels.insights` JSONB
- `useMemo` sur le parse

## H4. `apercu` page : sequential athlete → reports

**`app/(app)/athletes/[id]/apercu/page.tsx:54-56`** :
```ts
await supabase.from('athletes')... // #1
await Promise.all([reports, phases, ...]) // #2
```

**Fix** : tout dans un seul `Promise.all`. L'athlète est déjà dans `AthleteContext` de toute façon — `find(a => a.id === athleteId)` au lieu de re-fetcher.

## H5. `chart.js` eager sur `/bilans`

`app/(app)/athletes/[id]/bilans/page.tsx:14` → `BilanProgressView` → `MensurationCharts` → import direct chart.js.

**Fix** : `next/dynamic(() => import('./MensurationCharts'), { ssr: false })`. La page apercu fait déjà ça pour `WeightChart`.

## H6. `<img>` au lieu de `next/image` (15 fichiers)

1 seul fichier utilise `next/image` (AthletesList avatars). 15 utilisent `<img>` brut sans width/height/loading=lazy : avatars, bloodtest images, IG media, ContentPlanner.

**Fix** : remplacer par `<Image>` avec `loading="lazy"` (ou `priority` si above-fold).

**Impact** : WebP automatique + lazy + srcset → **30-50% gain** sur poids images.

## H7. `AthleteContext` — 21 colonnes pour 200 athlètes

`contexts/AthleteContext.tsx:27` : SELECT trop large. `complete_bilan_*` (5 cols), `water_goal_ml`, `pas_journalier` ne sont pas utilisés dans la liste — seulement sur les pages spécifiques.

**Fix** : SELECT minimal pour la liste + fetch détaillé à l'ouverture d'un athlète.

**Impact** : -50-100 KB payload sur login

## H8. `DashboardPage` re-renders cascade

- ligne 369 : useCallback deps `[sendingRappel, sentRappels, toast]` — Set mutables
- ligne 382 : `[user, toast]` au lieu de `[user?.id, toast]`
- ligne 405-410 : `mrr` filter+reduce sur athletes à chaque render sans useMemo
- ligne 487-510 : inline `style={{...}}` dans `.map()` — chaque render recrée 20+ refs

**Fix** : memoize tout ça. ~1h.

## H9. `daily_reports` no upper bound

**`components/bilans/BilansOverview.tsx:82-87`** : `.limit(200)` mais sans `lte(today)`. Si une row a une date corrompue (ex: `2099-12-31`), elle pollue les 200.

**Fix** : ajouter `.lte('date', toDateStr(new Date()))` partout.

## H10. RecorderContext deps cascade

`contexts/RecorderContext.tsx:248,273` : `stopRecording` capture 5 deps objet (recorder, user, toast, athleteIdForNext...). useEffect ligne 273 dépend de `stopRecording` → cascade re-subscribe.

**Fix** : extraire les setters en refs, primitiviser les deps.

# 🟡 IMPACT MOYEN

## M1. Stripe routes `select('*')` partout
`app/api/stripe/route.ts:449,579,817` + `cron:168` — bandwidth waste ~5-10 KB par lookup.

`app/api/stripe/route.ts:76-114` : 3-4 DB hits par request quand 1-2 suffisent.

## M3. Bloodtest extract sequential PDF fetch + base64
`app/api/bloodtest/extract/route.ts:158-174` : avec 3+ files, 100-300 ms cumulés. Parallel.

## M4. Annonces broadcast push sequential
`app/api/annonces/broadcast/route.ts:99-137` : insert notifs + push Expo en série.

## M5. Instagram messages no pagination
`app/api/instagram/messages/route.ts:37-65` : `select('*')` sans limit. Athlète 100+ msgs = full history.

## M6. Sidebar pas memoïsé
`components/layout/Sidebar.tsx` : monté sur toutes les routes (app), 12 icônes + hooks. Re-render à chaque nav.
**Fix** : `React.memo` + extract `navGroups` en const module-scope.

## M7. 114 console.log non-guardés en prod
Partout dans `app/`, `components/`. Pollution console + mineure perf hit.
**Fix** : helper `debug()` qui no-op en prod, ou ESLint rule.

## M8. Inter font 5 weights × 2 fichiers (latin)
**`app/layout.tsx:13`** : `weight: ['400', '500', '600', '700', '800']`. 5 woff2 latin + 5 latin-ext si fallback. ~150 KB de fonts.
**Fix** : revue UX — peut-être 3 weights suffisent (400, 600, 700) → ~90 KB économisés.

## M9. Pas de `<link rel="preload">` pour Inter
La page render avant que la font charge → FOIT (Flash of Invisible Text). 
**Fix** : `next/font/google` génère déjà des `preload` automatiquement, vérifier qu'il est bien généré.

## M10. `landing.module.css` (1401 lignes)
Importé dans `components/landing/*` qui est utilisé sur la page `/` (public landing). Pas chargé sur l'app authentifiée. ✓ pas de problème.

## M11. `training.module.css` (1014 lignes)
Importé sur `/athletes/[id]/training` uniquement. Beaucoup de selectors potentiellement non utilisés — audit avec `purgecss` ou DevTools Coverage tab.

## M12. `exercices` library — 500 rows client-filter
`app/(app)/exercices/page.tsx:100`, `ExerciseLibrary.tsx:35` : load 500, filter client-side. Pour un coach avec ~218 exos c'est OK, mais le pattern ne scale pas.
**Fix** : server-side filter par muscle si filtre actif.

## M13. SpeedInsights monté inline
`app/layout.tsx:36` : pas un gros impact (50ms environ).
